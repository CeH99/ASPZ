## Завдання 4.1: Максимальний обсяг пам'яті для malloc(3) [cite: 3]

**Теоретичне пояснення:**
Параметр `malloc(3)` є цілим числом типу даних `size_t`. Логічно, що максимальне число, яке можна передати — це максимальне значення `size_t` на платформі. У 64-бітній системі Linux `size_t` становить 8 байтів (64 біти). Відповідно, максимальний обсяг пам’яті, який може бути виділений за один виклик `malloc(3)`, дорівнює $2^{64}$ байт (або 16 ексабайт). 
Однак, теоретично максимальний обсяг складає 8 ексабайт, а не 16. Це зумовлено тим, що стандарт C вимагає, щоб різниця між двома вказівниками поміщалася у знаковий тип `ptrdiff_t` (максимум якого — $2^{63}-1$). Крім того, операційна система ділить віртуальний адресний простір навпіл між ядром та простором користувача.

**Файл з кодом:** `task4_1.c`

**Інструкція з компіляції та запуску:**
Щоб перевірити поведінку на x86_64 та x86, виконайте:
```bash
# Для 64-бітної архітектури (x86_64)
gcc 1.c -o task1_64
./task1_64

# Для 32-бітної архітектури (x86)
gcc -m32 1.c -o task1_32
./task1_32
```

### Пояснення поведінки програми:
В обох випадках malloc(3) повертає NULL. На 64-бітній системі система не може виділити 16 ЕБ пам'яті фізично та через обмеження простору користувача. На 32-бітній системі максимальне значення size_t становить 4 ГБ, але виділити суцільний блок такого розміру неможливо, оскільки частина адресного простору (зазвичай 1-2 ГБ) зарезервована під ядро ОС.

## Завдання 4.2: Від'ємний аргумент та переповнення

**Теоретичне пояснення:**
Функція `malloc(3)` очікує беззнаковий тип `size_t`. Якщо передати їй від’ємний аргумент, він буде неявно перетворений на величезне беззнакове число. 
У нашому тестовому випадку обчислюється кількість виділених байтів за формулою `num = xa * xb`[cite: 10]. Оскільки `num` оголошене як цілочисельна змінна зі знаком, результат множення великих чисел призводить до переповнення  (результат перевищує ліміт `int` і стає від'ємним). При передачі цього від'ємного значення `malloc(3)` інтерпретує його як запит на виділення колосального обсягу пам'яті, з яким система не може впоратися.

**Інструкція з компіляції та запуску:**
Програму потрібно запустити на x86_64 і x86:
```bash
# Для 64-бітної архітектури (x86_64)
gcc 2.c -o task2_64
./task4_2_64

# Для 32-бітної архітектури (x86)
gcc -m32 2.c -o task2_32
./task4_2_32
```
### Пояснення поведінки програми:
В обох архітектурах malloc(3) поведе себе безпечно і поверне NULL. Через переповнення int значення стає від'ємним. На x86_64 воно перетворюється на значення близьке до 16 ЕБ, а на x86 — близьке до 3-4 ГБ. В обох випадках ОС відмовляє у виділенні такого цілісного шматка пам'яті.

## Завдання 4.3: Виклик malloc(0)

**Теоретичне пояснення:**
Згідно зі стандартом мови C, поведінка `malloc(0)` залежить від конкретної реалізації (implementation-defined). Функція може або повернути `NULL`, або повернути унікальний вказівник, який згодом можна безпечно передати у функцію `free()`. 
У стандартній бібліотеці `glibc` (яка використовується в більшості дистрибутивів Linux), `malloc(0)` зазвичай повертає валідний вказівник на мінімальний можливий блок пам'яті (зазвичай 16 або 32 байти через необхідність зберігання метаданих та вирівнювання), але доступний для користувача розмір цього блоку дорівнює нулю. Писати дані за цією адресою не можна (це призведе до невизначеної поведінки), але цей вказівник обов'язково треба звільнити через `free()`, щоб уникнути витоку пам'яті.

**Інструкція з компіляції та запуску:**
Для компіляції та запуску програми з використанням утиліти `ltrace` (яка перехоплює виклики динамічних бібліотек), виконайте наступні команди:

```bash
gcc 3.c -o task3
ltrace ./task3
```

### Пояснення поведінки програми (виводу ltrace):
Під час виконання через ltrace можна чітко побачити, як програма звертається до бібліотеки C.
У виводі буде зафіксовано виклик malloc(0), який повертає певну адресу в пам'яті (наприклад, 0x...). Відразу після цього викликається функція free() з цією ж адресою. Це підтверджує, що glibc дійсно виділяє структуру на купі (heap) навіть для нульового розміру, і система коректно її обробляє при звільненні.

## Завдання 4.4: Помилки в циклі з malloc та free

У наведеному фрагменті коду запитується, чи є помилки в такому циклі:
`void *ptr = NULL;`
`while (<some-condition-is-true>) {`
`if (!ptr)` 
`ptr = malloc(n);` 
`[... <використання 'ptr'> ...]`
`free(ptr);` 
`}`

У цьому коді присутні дві критичні помилки доступу до пам'яті, які проявляться на другій ітерації циклу: **Use-After-Free** (використання після звільнення) та **Double Free** (подвійне звільнення). 
Після першої ітерації циклу функція `free(ptr)` звільняє пам'ять[cite: 23], але сам вказівник `ptr` не занулюється. Він залишається "висячим" (dangling pointer).На наступній ітерації умова `if (!ptr)` виявиться хибною, нова пам'ять не виділиться, і програма спробує працювати зі звільненою пам'яттю, а потім вдруге її звільнити. Це призведе до краху програми.

**Інструкція з компіляції та запуску:**
Щоб продемонструвати проблему та правильний варіант коду, виконайте:
```bash
gcc 4.c -o task4
./task4
```

## Пояснення поведінки програми:
Щоб уникнути краху, одразу після виклику free(ptr)  необхідно зануляти вказівник (ptr = NULL;). Завдяки цьому на наступній ітерації умова if (!ptr) знову стане істинною , malloc(n) коректно виділить новий блок пам'яті, і програма працюватиме стабільно. Наш тестовий випадок успішно відпрацьовує виправлену логіку для кількох ітерацій.

## Завдання 4.5: Поведінка realloc(3) при нестачі пам'яті

**Теоретичне пояснення:**
Якщо функція `realloc(3)` не може виділити новий обсяг пам'яті, вона повертає `NULL`. Найважливіше правило полягає в тому, що оригінальний блок пам'яті при цьому не звільняється і залишається дійсним. 
Тому використання конструкції `ptr = realloc(ptr, new_size);` є небезпечним: якщо `realloc` завершиться невдало, змінна `ptr` стане `NULL`, і посилання на початковий блок пам'яті буде втрачено. Це класичний приклад витоку пам'яті (memory leak). Щоб цього уникнути, результат `realloc` завжди потрібно записувати у тимчасову змінну і перевіряти на `NULL`.

**Інструкція з компіляції та запуску:**
```bash
gcc 5.c -o task5
./task5
```

### Пояснення поведінки програми:
Програма спочатку успішно виділяє 1 КБ пам'яті. Потім вона намагається змінити розмір цього блоку до максимально можливого значення size_t (близько 16 ЕБ на 64-бітних системах) за допомогою realloc. Оскільки система не може задовольнити цей запит, realloc повертає NULL. Програма демонструє правильну обробку цієї ситуації: вона перехоплює помилку через тимчасовий вказівник і коректно звільняє оригінальний блок пам'яті (original_ptr), уникаючи витоку.

## Завдання 4.6: Поведінка realloc(3) з NULL та розміром 0

**Теоретичне пояснення:**
Функція `realloc(3)` має особливу поведінку для граничних значень аргументів:
1. Виклик `realloc(NULL, size)` є еквівалентним виклику `malloc(size)`. Оскільки старого блоку для зміни розміру немає, система просто виділяє новий блок заданого розміру.
2. Виклик `realloc(ptr, 0)`, де `ptr` — дійсний вказівник, у стандартній бібліотеці `glibc` працює як `free(ptr)`. Пам'ять звільняється, і функція зазвичай повертає `NULL`.

**Інструкція з компіляції та запуску:**
```bash
gcc 6.c -o task6
./task6
```

### Пояснення поведінки програми:
У тестовому випадку ми спочатку викликаємо realloc(NULL, 1024). Програма успішно виділяє пам'ять і виводить отриману адресу, підтверджуючи, що функція відпрацювала як malloc.
Потім ми передаємо цю ж адресу в realloc(ptr, 0). Функція звільняє пам'ять і повертає NULL, демонструючи поведінку, аналогічну до free. Це зручна властивість, яка дозволяє використовувати realloc як універсальну функцію для управління пам'яттю (виділення, зміна розміру та звільнення) в деяких алгоритмах.

## Завдання 4.7: Використання reallocarray(3)

**Теоретичне пояснення:**
[cite_start]Оригінальний код використовував `realloc(ptr, 500 * sizeof(struct sbar))`[cite: 34]. Головний недолік такого підходу — ризик цілочисельного переповнення при множенні аргументів (якщо числа великі). Якщо множення переповниться, `realloc` виділить замало пам'яті, що може призвести до краху програми або вразливості безпеки.
[cite_start]Функція `reallocarray(ptr, nmemb, size)` вирішує цю проблему. Вона виконує множення `nmemb * size` безпечно: у разі переповнення вона одразу повертає `NULL`, навіть не намагаючись виділити пам'ять. Це робить код надійнішим.

**Інструкція з компіляції та запуску:**
Щоб порівняти виклики, ми використовуємо утиліту `ltrace`:
```bash
gcc 7.c -o task7
ltrace ./task7
```

### Порівняння результатів ltrace:
Якби ми використовували звичайний realloc, у виводі ltrace ми б побачили щось на кшталт realloc(0x..., 20000), оскільки компілятор (або процесор) перемножив би 500 * 40 ще до виклику функції. Бібліотека C не знала б, з чого складається число 20000.
З reallocarray вивід ltrace демонструє іншу картину: ми чітко бачимо виклик reallocarray(0x..., 500, 40) (де 40 — це розмір структури sbar у байтах). Відповідальність за безпечне множення і виділення перекладається на саму функцію бібліотеки C (glibc), яка має вбудовані перевірки на переповнення.

## Індивідуальне завдання: Порівняння calloc() та malloc()+memset() (Варіант 8)

**Теоретичне пояснення та вплив zero-page:**
На перший погляд, `calloc(1, SIZE)` і `malloc(SIZE) + memset(ptr, 0, SIZE)` роблять одне й те саме: виділяють блок пам'яті і заповнюють його нулями. Однак під капотом операційна система (ОС) обробляє їх зовсім по-різному для великих блоків пам'яті.

Коли ми запитуємо великий обсяг пам'яті (наприклад, 1 ГБ), `glibc` використовує системний виклик `mmap` замість розширення купи (brk). 
1. **Як працює `calloc` (Ліниве виділення та zero-page):**
   `calloc` знає, що пам'ять має бути заповнена нулями. ОС застосовує оптимізацію: вона не виділяє 1 ГБ фізичної оперативної пам'яті і не записує туди нулі. Замість цього вона мапить (відображає) всі віртуальні сторінки цього гігабайта на одну-єдину спеціальну фізичну сторінку ОС — **zero-page** (сторінку, повністю заповнену нулями). Ця сторінка доступна лише для читання. Тому `calloc` повертає результат майже миттєво. Справжня фізична пам'ять буде виділятися посторінково (через page fault) лише тоді, коли програма спробує *записати* дані за цими адресами.
2. **Як працює `malloc` + `memset`:**
   `malloc` так само мапить віртуальну пам'ять. Але коли ми одразу викликаємо `memset(ptr, 0, SIZE)`, ми примусово робимо запис у кожен байт цього гігабайта. ОС змушена обробляти мільйони сторінкових переривань (page faults), виділяти реальні фізичні сторінки оперативної пам'яті та фізично записувати туди нулі. Це вимагає значних витрат процесорного часу та пропускної здатності шини пам'яті, тому цей підхід працює у сотні або навіть тисячі разів повільніше.

**Інструкція з компіляції та запуску:**
Оскільки ми використовуємо функцію `clock_gettime`, на старих системах може знадобитися лінкування бібліотеки часу (хоча у сучасних дистрибутивах Linux це зазвичай не потрібно).
```bash
gcc MyTask.c -o mytask
./mytask
```

### Результати вимірювання часу:
При запуску програми calloc виконується за долі мілісекунди (наприклад, ~0.000010 сек), оскільки він лише налаштовує таблиці сторінок віртуальної пам'яті. Натомість malloc + memset займає відчутний час (наприклад, ~0.2 - 0.5 секунд залежно від швидкості оперативної пам'яті комп'ютера), оскільки процесор фізично записує 1 ГБ нулів в ОЗП.
